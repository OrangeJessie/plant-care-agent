# 项目报告书 — Plant Care Agent（种田宝）

## 1. 项目概述

**项目名称：** Plant Care Agent（种田宝）

**目标：** 基于 NVIDIA NeMo Agent Toolkit（NAT）构建一个覆盖「家庭植物养护」与「农场规模管理」的全流程 AI 助手。在个人版中，它不仅是养护工具，更是一个懂你的「植物陪伴 AI」——记录每一天的生长变化、主动推送巡检日报、用时间线和看板讲述植物的成长故事；在农业版中，它是多地块传感器数据与操作决策的智能中枢。

**背景与要解决的实际问题：**

植物管理覆盖从阳台到农场的广阔场景，个人养花与规模种植各有各的痛点：

**个人场景**

1. **情感陪伴缺失**：城市生活节奏快，越来越多人渴望陪伴感——"养芒狗"等虚拟宠物的流行就是印证。养宠物门槛高，养植物虽然容易，却因为植物"不会说话"、生长变化缓慢，很难建立持续的情感连接。**人们需要的不只是一个浇水提醒工具，而是一个能把植物的每一天变化「看见」并「说出来」的伙伴**。

2. **知识碎片化**：养花人常在社交媒体、论坛、短视频中零散获取信息，缺乏系统性和个性化。面对具体问题（如"栀子花叶子发黄是什么原因"）往往无法获得结合当前天气、养护历史、生长阶段的精准建议。

3. **管理不持续**：植物养护需要长期坚持浇水、施肥、修剪等操作，但多数人缺乏结构化记录手段。养护日程难以结合实时天气和节气变化动态调整，容易出现"忘了浇水"或"节气该施肥了但不知道"的情况。

4. **问题发现滞后**：植物出现病虫害、缺水等问题时，往往已经到了肉眼可见的程度才被发现。缺乏主动巡检和提前预警机制，尤其是极端天气来临前无法及时采取保护措施。

**农业场景**

5. **传感器数据看不过来**：一个地块部署土壤湿度、温度、pH、光照等多类传感器，多地块叠加后数据量激增。农户既无法 24 小时盯仪表盘，也难以从海量数值中判断"哪个指标异常、该怎么处理"——缺少一个自动读取、分析、报警的智能中间层。

6. **操作决策凭经验、缺追溯**：灌溉量多少、施肥时机怎么选、除虫用什么药——高度依赖个人经验，缺乏数据支撑。操作执行后也没有结构化记录，无法回溯"上周 A 地块施了什么肥、效果如何"，经验无法沉淀复用。

7. **多地块协调成本高**：不同地块种植品种、生长阶段、土壤条件各异，需要差异化管理。农户只能逐块巡查、手动记录，地块一多就容易遗漏和混淆。

**两个场景看似不同，但底层逻辑一致——都是"观察 → 分析 → 决策 → 行动 → 记录"的循环。** Plant Care Agent 正是基于这一洞察，用同一个 AI Agent 框架统一覆盖两个场景：在个人版中，它将植物的生长过程搬进虚拟世界，用成长档案和主动推送让养植物像养宠物一样有陪伴感；在农业版中，它接管传感器数据流和操作链条，成为农场的智能管家。用户只需用自然语言对话，就能完成从养护到管理的全部工作。

## 2. 作品介绍（亮点 / 特点 / 功能）

### 2.1 核心亮点

#### 亮点一：虚拟植物陪伴 — 让养植物像养宠物一样有温度

针对上述"情感陪伴缺失"的痛点，**Plant Care Agent 把植物的生长过程搬进了虚拟世界**，让每棵植物都变成一个会"说话"的数字伙伴：

- **成长故事档案**：AI 为每棵植物建立 Markdown 日志，从播种到开花，每一次浇水、施肥、修剪、状态变化都被记录成"成长故事"，用户可以随时回溯"我的栀子花这两周经历了什么"。
- **主动日报推送**：不需要你去看它——Agent 每天在你设定的时间主动推送巡检日报到手机，就像宠物到点了来找你一样，告诉你"今天栀子花状态不错，新叶展开了"或"注意明天有寒流，建议搬进室内"。
- **时间线与看板可视化**：通过 `timeline`、`dashboard`、`compare` 工具，把枯燥的养护记录变成生长时间线、多维健康看板和多植物横向对比，让生长过程"可见"。
- **对话即陪伴**：随时对 Agent 说"我的栀子花最近怎么样？"、"该施肥了吗？"，Agent 结合天气、季节、养护历史给出有温度的回应——不是冷冰冰的数据，而是一个懂你植物的朋友。

这是将植物养护从"工具型"升维为"陪伴型"的核心差异点。

#### 亮点二：双模式架构 — 同一个 Agent，从阳台长到农场

个人养护和农业管理看似是两个世界，但底层是同一套"观察→分析→决策→行动→记录"的循环。Plant Care Agent 没有做两个独立产品，而是用**同一套 Agent 框架 + 不同工具集**统一覆盖两个场景，通过配置文件一键切换：

| 模式 | 场景 | 核心能力 | 对应痛点 |
|------|------|----------|---------|
| **个人版（种田宝·个人版）** | 家庭阳台、花园 | 植物陪伴、成长故事、养护日程、图像诊断、定时巡检推送 | 情感缺失、知识碎片、管理不持续、发现滞后 |
| **农业版（种田宝·农业版）** | 规模化种植 | 7 类传感器监控、自动化操作（灌溉/施肥/除虫）、地块日志与报告 | 传感器过载、决策缺追溯、多地块协调难 |

两个版本**共享**天气预报、联网搜索、图像分析、Skills 系统、主动巡检推送等基础能力——个人版中积累的养护知识和交互经验，在农业版中同样适用；而农业版中验证的传感器分析和自动化决策能力，未来也可以反哺个人用户（如接入家庭土壤湿度传感器）。这种"同根生长"的架构让两个版本互相增强，而非各自为战。

#### 亮点三：主动巡检与推送 — AI 不等你问，主动来报告

传统 Agent 只在用户提问时响应。Plant Care Agent 具备**主动巡检体系**，且这套机制横跨个人版与农业版——个人版下它是"每天来找你汇报的植物伙伴"，农业版下它是"7×24 值守的农场管家"：

- **每日定时巡检**：Agent 服务启动后自动运行后台任务，在用户指定时间（如每天 21:17）对所有植物/地块执行全面巡检（天气风险 + 养护状态 + 生长进展 + 病虫害风险 + 传感器异常），并通过 ntfy / webhook **主动推送结果到手机**。
- **紧急天气告警**：每 2 小时轮询天气预报，当检测到冰雹、冰冻、强降水、大风等 critical 天气时，立即推送预警（24 小时内去重）。对个人用户而言是"明天降温，搬进室内"；对农场而言是"霜冻预警，启动地膜覆盖"。
- **用户可在对话中配置**：只需说"设置每天 9 点自动巡检"，Agent 自动完成配置。

#### 亮点四：可组合的 Skills 系统 — 类 Claude Code 的技能加载机制

借鉴 Claude Code / Cursor 的 Skill 设计理念，实现了**目录化 Skills 系统**：

- 每个 Skill 是一个目录 + `SKILL.md` 文件，YAML frontmatter 描述名称和触发条件
- 支持 4 层路径覆盖（包内置 → 环境变量 → 项目根 → 用户目录）
- Agent 运行时自动注入 Skills 目录摘要，按需 `load_skill` 加载完整工作流
- 内置 8 个 Skills：植物巡检、病虫诊断、阳台种菜、灌溉管理、作物轮作、IPM 虫害管理等

#### 亮点五：7 类物理模型传感器模拟

农业版内置完整的**传感器模拟子系统**，每类传感器均基于真实物理模型：

| 传感器 | 物理模型 | 操作影响 |
|--------|---------|---------|
| 土壤湿度 | 蒸散（ET）衰减 + 降雨补充 + 灌溉指数吸收 | 灌溉 → 湿度跃升 |
| 空气温度 | 日正弦变化 + 天气 API 耦合 + 云量阻尼 | 通风遮阳 → 降温 |
| 空气湿度 | 温度负相关 + 夜间露点 + 降雨增益 | 灌溉微增，通风降低 |
| 光照强度 | 太阳高度角 + 云量 + 季节因子 | — |
| 土壤 pH | 土壤类型基线 + 缓慢回复动力学 | 施肥 → pH 偏移 |
| 风速 | Weibull 分布 + 阵风概率事件 | — |
| 降雨量 | 降水概率门控 + Gamma 强度采样 | — |

传感器之间存在交叉耦合（如温度影响湿度和蒸散速率），操作执行（灌溉、施肥等）会实时反馈到传感器数值，形成闭环。

#### 亮点六：多模态植物诊断

支持用户上传植物照片，调用本地多模态模型（如 LLaVA）进行图像分析，结合植物知识库给出病虫害诊断和处理建议。

## 3. 技术创新点

### 创新一：巡检双通道设计 — LLM 灵活分析 + 规则引擎可靠兜底

| 通道 | 使用场景 | 实现方式 | 优势 |
|------|---------|---------|------|
| **LLM 工具通道** | 用户对话时 | LLM 调用 4 个数据工具获取原始数据，自主分析 | 灵活、可结合植物知识、给个性化建议 |
| **规则引擎通道** | 后台定时/紧急 | PlantInspector 硬编码规则引擎 | 无需 LLM 调用、低延迟、零成本、适合高频自动化 |

两个通道覆盖不同场景：对话时由 LLM 做深度分析（利用植物知识和上下文），后台巡检由规则引擎保证可靠性和低成本。

### 创新二：Prompt 分层组装与注入机制

系统提示词采用**多段 Markdown 文件分层组装**的架构：

```
prompts/
  core.md         → 角色定义 + 核心指令
  workflow.md      → 工具列表 + 工作流程
  style.md         → 回答风格与格式要求
  tools_block.md   → ReAct 格式规范（含 {tools} 占位符）
```

运行时由 `assemble_config.py` 将 Markdown 内容拼接并注入到 NAT 的 config YAML 中，实现**提示词的版本化管理与模块化复用**。同时 Wrapper 层在每次请求时动态注入植物日志摘要、巡检结果、Skills 索引等上下文信息。

### 创新三：传感器物理模型与操作闭环

传感器不是简单的随机数生成器，而是基于真实物理模型的模拟器：

- **蒸散模型**（ET）：土壤湿度随时间因蒸散作用线性下降，速率受温度和风速影响
- **日变化正弦**：空气温度遵循以 14 时为峰值的正弦日变化曲线
- **交叉耦合**：温度影响湿度、降雨影响土壤湿度、云量影响光照
- **操作反馈闭环**：执行灌溉后土壤湿度传感器立即响应（指数吸收曲线），施肥后 pH 偏移并缓慢回复

这使得演示中传感器数据具有真实感，操作决策有据可循。

### 创新四：记忆系统 — 种植日志 + 对话记忆双轨持久化

- **种植日志**：每棵植物独立文件夹，Markdown 格式的结构化日志（YAML frontmatter + 按日期分组的事件），支持 LLM 读取分析
- **对话记忆**：按 User + Session 持久化到 Markdown 文件，支持增量/全量两种同步模式
- **Wrapper 注入**：每次对话前自动读取并注入关注植物的完整日志 + 其他植物的摘要，让 Agent 具备"记忆"

### 创新五：模块化 Skills 系统

参考 Claude Code 和 Cursor 的 Skill 机制，实现了生产级的 Skills 系统：

- 纯 Markdown 定义，无需编写代码即可扩展 Agent 能力
- 四层路径覆盖机制，支持内置 → 项目级 → 用户级的灵活扩展
- Agent 运行时自动注入可用 Skills 索引，按需加载避免 token 浪费

## 4. NVIDIA 工具平台 SDK 与开源模型使用

### NVIDIA NeMo Agent Toolkit（NAT）

项目**完全基于 NAT 框架构建**，深度使用了以下核心能力：

| NAT 能力 | 项目中的应用 |
|----------|------------|
| **`register_function`** | 注册了 27+ 个 NAT 工具函数（植物知识、天气预报、生长日志、传感器监控、自动化操作等） |
| **`register_function_group`** | 农场日志等多工具组的统一注册（FunctionGroup） |
| **`FunctionBaseConfig`** | 每个工具通过 Pydantic 配置类定义参数，config.yml 统一管理 |
| **`react_agent`** | 内层 Agent 使用 NAT 的 ReAct Agent，支持多轮工具调用 |
| **`Workflow 机制`** | 自定义 `plant_memory_wrapper` / `farm_memory_wrapper` 作为 Workflow 入口，实现请求前注入记忆 + 巡检 |
| **`nat serve`** | 使用 NAT 的 FastAPI 服务框架启动 Agent 服务 |
| **`Object Store`** | 使用 NAT 的对象存储支持图片上传（plant_image_analyzer） |
| **`LLM 抽象`** | 通过 NAT 的 LLM 配置统一管理模型接入（支持 Ollama / OpenAI 兼容 API） |

### 模型接入

通过 NAT 的 LLM 抽象层统一管理模型接入，只需修改配置文件即可在两种方式间切换：

| 接入方式 | 适用场景 | 优势 | 取舍 |
|---------|---------|------|------|
| **本地部署**（Ollama 等） | 有 GPU 服务器、对数据隐私要求高 | 数据不出本机，零调用成本，无网络依赖，离线可用 | 需要硬件投入，模型规模受显存限制 |
| **外部 API**（OpenAI 兼容接口） | 无 GPU 或追求更强模型能力 | 零部署成本，可调用最新大参数模型，推理速度稳定 | 数据经过网络传输，按调用量计费 |

用户可根据自身条件灵活选择：注重隐私和成本可控选本地部署，追求模型能力上限选外部 API，也可以混合使用（如主对话走本地、图像诊断走 API）。

## 6. 未来展望

1. **真实物联网对接**：当前传感器为物理模型模拟，未来计划接入 MQTT / LoRaWAN 等真实物联网传感器协议，实现从模拟到真实数据的无缝切换，支持实际农业部署。

2. **个人版与农业版深度融合**：让个人用户也能接入家庭级传感器（如蓝牙土壤湿度探针），享受农业版的数据驱动决策能力；同时将个人版的"陪伴式交互"和"成长故事"引入农业场景，为每个地块生成可读的"生长叙事报告"，让农场管理也有温度。

3. **更强的视觉诊断能力**：引入更专业的植物病害识别模型（如基于 NVIDIA TAO 训练的专用视觉模型），实现对常见病虫害的高精度自动分类与严重程度评估。

4. **多人协作与共享农场**：支持多用户查看同一花园/农场的数据，实现家庭成员共同管理或农场团队协作。增加权限控制与操作审批流程。

5. **智能养护日程优化**：结合长期天气预报与植物生长周期模型，自动优化浇水/施肥时间窗口。引入强化学习，根据历史养护效果持续改进推荐策略。

6. **移动端与语音交互**：开发微信小程序或 App 前端，支持拍照识别、语音对话、推送通知的一体化体验，进一步降低使用门槛。

7. **NVIDIA DGX 云端部署**：利用 NVIDIA DGX Cloud 部署更大参数量模型（如 70B+），提升对话质量与诊断准确性；同时利用 GPU 加速批量传感器数据分析与预测。
